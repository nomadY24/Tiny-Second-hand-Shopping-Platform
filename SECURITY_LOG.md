# 보안 약점 발견 및 수정 기록

## SEC-001 비밀번호 평문 저장 위험

- 최초 위험: 입력 비밀번호를 그대로 저장하면 DB 유출 즉시 계정이 탈취된다.
- 원인: 인증 데이터의 민감도를 고려하지 않은 저장 방식.
- 수정: Werkzeug `generate_password_hash`로 솔트 포함 단방향 해시만 저장하고 `check_password_hash`로 검증한다.
- 검증: `test_register_hashes_password`에서 DB 값이 원문과 다르고 올바른 암호만 검증되는지 확인했다.

## SEC-002 타인 상품 수정·삭제(IDOR)

- 최초 위험: URL의 상품 번호만 변경해 다른 사용자의 상품을 수정·삭제할 수 있다.
- 원인: 화면 버튼 숨김에 의존하고 서버 소유권 검사를 누락할 가능성.
- 수정: 모든 수정·삭제 라우트에서 판매자 ID와 현재 사용자 ID를 비교하고 관리자만 예외로 허용한다.
- 검증: `test_idor_owner_check_blocks_edit`에서 타 사용자 요청이 HTTP 403인지 확인했다.

## SEC-003 음수·과다 금액 입력

- 최초 위험: 음수 가격/송금액이 잔액 증가나 잘못된 데이터로 이어질 수 있다.
- 원인: 클라이언트 입력을 신뢰하는 설계.
- 수정: WTForms 범위 검증과 DB `CHECK(amount > 0)` 제약을 적용하고 잔액을 서버에서 재확인한다.
- 검증: 음수 가격, 음수 송금, 잔액 초과 송금 테스트를 수행했다.

## SEC-004 중복 송금

- 최초 위험: 새로고침 또는 요청 재전송으로 동일 송금이 두 번 처리될 수 있다.
- 원인: 요청을 구별하는 식별자가 없었다.
- 수정: 폼 생성 시 무작위 멱등키를 만들고 `Transfer.idempotency_key`에 UNIQUE 제약을 설정했다. 잔액 변경과 거래 기록은 하나의 DB 커밋으로 처리한다.
- 검증: 동일 멱등키 요청을 두 번 보내 거래가 한 건만 저장되는지 확인했다.

## SEC-005 저장형 XSS

- 최초 위험: 상품명·설명·메시지에 스크립트가 저장되어 다른 사용자의 브라우저에서 실행될 수 있다.
- 원인: 사용자 HTML을 신뢰하거나 `safe` 렌더링을 사용할 때 발생한다.
- 수정: Jinja 자동 이스케이프를 유지하고 사용자 입력에는 `safe` 필터를 사용하지 않는다. 길이도 제한한다.
- 검증: 상품명에 `<script>`를 저장한 뒤 HTML 엔티티로 출력되는지 확인했다.

## SEC-006 CSRF

- 최초 위험: 로그인 사용자가 공격자 페이지를 열었을 때 상품 삭제·송금·관리자 조치가 대신 실행될 수 있다.
- 원인: 상태 변경 요청에 사용자 의도를 증명하는 토큰이 없다.
- 수정: Flask-WTF CSRF 보호를 전역 활성화하고 모든 POST 폼에 토큰을 포함했다. 삭제와 로그아웃도 GET이 아닌 POST로 구현했다.
- 검증: 토큰 없는 회원가입 POST가 HTTP 400으로 거부되는지 확인했다.

## SEC-007 관리자 권한 상승

- 최초 위험: 일반 사용자가 관리자 URL을 직접 호출할 수 있다.
- 원인: 메뉴 표시 여부와 서버 권한 검사가 혼동될 수 있다.
- 수정: `admin_required` 데코레이터가 모든 관리자 라우트에서 인증과 역할을 검사한다.
- 검증: 일반 사용자로 `/admin` 요청 시 HTTP 403을 확인했다.

## SEC-008 중복 신고 및 자기 신고

- 최초 위험: 한 사용자가 신고 수를 인위적으로 늘리거나 자신을 신고해 데이터 품질을 훼손할 수 있다.
- 원인: 신고 조합의 고유성 및 의미 검증이 없다.
- 수정: `(reporter_id, target_type, target_id)` UNIQUE 제약, 대상 존재 확인, 자기 사용자 신고 금지를 적용했다.
- 검증: 같은 상품을 두 번 신고해 신고가 한 건만 남는지 확인했다.

## SEC-009 사용자 식별 정보 노출

- 최초 위험: 로그인 실패 메시지가 계정 존재 여부를 알려 사용자 이름 열거를 돕는다.
- 원인: "없는 아이디"와 "틀린 비밀번호"를 구분해서 보여준다.
- 수정: 두 경우 모두 동일한 일반 오류 메시지를 반환한다.
- 검증: 존재/부재 아이디에 같은 메시지가 반환되는지 확인했다.

## SEC-010 내부 오류 정보 노출

- 최초 위험: 스택 추적, 파일 경로, DB 구조가 사용자 화면에 표시될 수 있다.
- 원인: 디버그 모드 또는 기본 예외 페이지 사용.
- 수정: 디버그를 끄고 400/401/403/404/500 사용자용 오류 페이지를 등록했다.
- 검증: 권한·CSRF 오류가 내부 세부정보 없이 사용자 메시지로 표시되는지 확인했다.
